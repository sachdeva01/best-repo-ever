#### вЪ†пЄП Risk 1: `GET /api/dashboard/quick-stats` вАФ Inline `yfinance` calls in the hot path
**File:** `backend/routers/dashboard.py` вЖТ `get_quick_stats()`

```python
# Inside a loop over every ETF in TARGET_ALLOCATION:
current_yield = get_current_yield(symbol)  # calls get_current_yield() from portfolio_allocation
```

`get_current_yield()` currently returns static values from `STATIC_YIELDS` (making it fast), but if the 24-hour TTL expires and `yfinance` is actually invoked, each `yf.Ticker(symbol).info` call can take **1вАУ3 seconds per symbol**. There are **9 symbols** in `TARGET_ALLOCATION`. Under the worst-case cold-cache scenario, this single endpoint could take **9вАУ27 seconds**.

**Fix:** The background-task price refresh pattern already exists in `portfolio_allocation.py` вАФ apply the same pattern to yields. Use `BackgroundTasks` to refresh the yield cache off the request path, and always serve from the in-memory `_yield_cache` dict synchronously:

```python
# dashboard.py вАФ quick-stats should never trigger a live yfinance fetch inline
# Move yield refresh to a background task exactly like _refresh_prices_background()
@router.get("/dashboard/quick-stats")
async def get_quick_stats(background_tasks: BackgroundTasks, db: Session = Depends(get_db)):
    if _yield_cache_needs_refresh():
        background_tasks.add_task(_refresh_yields_background)
    # rest of handler reads only from cache, never blocks
```

#### вЪ†пЄП Risk 2: `GET /api/dashboard/quick-stats` вАФ Year-loop projection computed on every request
**File:** `backend/routers/dashboard.py` вЖТ `get_quick_stats()`

```python
balance = total_net_worth
for year in range(years_to_withdrawal_calc):  # up to ~40 iterations
    balance += annual_reinvestment
    balance *= (1 + expected_growth_rate)
```

This is a simple loop, but it also calls `calculate_expected_growth_rate(db)` which fires a DB query on every request. The result is a constant (`0.06`) вАФ it never changes.

```

```

### 4. Issues Found & Recommended Fixes

#### вЪ†пЄП Issue 1 вАФ Stale Fallback Values in Error Handler (Low severity)

**Location:** `market_data.py`, `fetch_external_market_data()`, lines ~55вАУ60

**Problem:** If yfinance fails (network outage, Yahoo ban), the router silently returns hardcoded fallback values (`S&P 500: 5200.0`, `Nasdaq: 16500.0`, `Treasury: 4.25`) and persists them to the database as if they were live. These values are significantly outdated and could mislead users without any warning.

**Fix:**
```python
# Instead of silently persisting fallback values, raise an HTTPException
# so the caller can handle it gracefully, OR log a warning and skip the DB write.

except Exception as e:
    print(f"[WARNING] yfinance fetch failed for {data_type}: {e}")
    # Return None and let the router decide whether to use cached DB value
    return None
```
Then in the router, only write to DB if the returned value is not `None`, otherwise serve the last valid cached row.
